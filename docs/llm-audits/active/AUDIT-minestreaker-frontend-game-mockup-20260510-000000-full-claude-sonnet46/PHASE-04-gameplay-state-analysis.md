## 1. State Creation

### Board State Creation
- `Board(width, height, mine_positions)` — creates all numpy arrays, computes neighbours via Python loop
- `place_random_mines()` — uses `np.random.default_rng(seed)` for reproducible placement
- `load_board_from_npy()` — loads and validates neighbour consistency
- `load_board_from_pipeline()` — BROKEN (FIND-ARCH-CRITICAL-f003a, f004a)

### First-Click Safety
- `GameEngine._first_click = True` guards safe-first behavior
- On first `left_click(x, y)`: if the clicked cell is a mine, board is **regenerated** with `seed+1`
- This regeneration creates a **new Board object** — the engine's board reference is replaced
- **Issue**: regeneration is correct but creates 2 boards (one discarded) with no cleanup notification to Renderer

## 3. State Synchronization Issues

### 3.1 Renderer holds stale `self.board` reference
- `Renderer.__init__` stores `self.board = engine.board`
- After first-click regeneration, `engine.board` is replaced with a new Board object
- `self.board` in Renderer still points to the OLD board
- **Impact**: Renderer renders stale cell data after first-click on mine cell

**Evidence**: `gameworks/renderer.py` line ~255: `self.board = engine.board` (stored at init, not re-read)
**Evidence**: `gameworks/engine.py` ~line 282: `self.board = Board(board.width, board.height, mp)` (board replaced)

### 3.2 No board reference update path
When `GameEngine.left_click()` regenerates the board, there is no mechanism to notify the Renderer that `engine.board` has changed. The Renderer's cached `self.board` reference becomes stale and diverges from truth.

### 3.3 MoveResult.flagged semantics
`MoveResult.flagged` is typed as `bool` but `Board.toggle_flag()` returns `str` ('flag', 'question', 'hidden'). In `right_click()`: `return MoveResult(flagged=placed, ...)` where `placed` is a string. Python's truthy coercion means `flagged='flag'` → True, `flagged='hidden'` → True (non-empty string is truthy). The semantics are incorrect — `flagged=True` even when the flag was removed.

## 4. Save/Load State Correctness

### Save Correctness
- `_save_npy()` saves mine=-1, else neighbour_count
- Correctly reconstructs the board layout
- **Issue**: Saves the INITIAL board state (mine layout only), not the CURRENT game state (revealed cells, flags). Loading a saved board always starts from the beginning — no save/resume.

### Load Correctness
- `load_board_from_npy()` validates neighbour counts against computed values
- Raises `ValueError` on mismatch — good defensive validation
- `_build_engine()` creates a GameEngine in random mode then replaces the board — inconsistent: `self.mode` will be "random" but board is from NPY

### Pipeline .npy Format Incompatibility (FIND-ARCH-CRITICAL-f006a) — NEW

The three boards in `results/iter9/` were generated by `run_iter9.py` which saves `int8` grids with `0=safe, 1=mine` (raw SA optimization output). `load_board_from_npy()` detects mines with:

```python
if grid[y, x] < 0:   # ← never true for pipeline boards
    mine_pos.add((x, y))
```

Since `1 >= 0`, all mine cells pass through as safe. The loaded `Board` will have:
- `mine_positions = set()` — zero mines
- `total_mines = 0`
- Win condition immediately satisfied on first cell reveal
- Neighbour validation step will raise `ValueError` (computed neighbours=0, stored=0 after save) or silently pass — depends on whether the pipeline board also has correct neighbour counts stored

**Impact on the three committed boards**:
| Board | Expected Mines | Actual Loaded Mines |
|---|---|---|
| 300×370 (14.0% density) | 15,574 | 0 |
| 300×215 (28.7% density) | 18,529 | 0 |
| 600×429 (24.5% density) | 63,060 | 0 |

The neighbour validation in `load_board_from_npy()` will also fail because the pipeline format stores raw `0/1` values, not neighbour counts — the validator will detect the mismatch and raise `ValueError`, so these boards cannot load at all in the current implementation.

## 5. Determinism

### Seeded Random Boards
- `place_random_mines(width, height, count, safe_x, safe_y, seed=seed)` uses `np.random.default_rng(seed)`
- Same seed → identical mine positions → fully deterministic
- First-click safety uses `seed+1` for regeneration — deterministic for a given seed+click combo

### SA-Based Boards (Image Mode)
- SA kernel uses `np.random.seed(seed)` inside Numba JIT — deterministic given same Numba version
- However, parallel candidate evaluation in `repair.py` uses `ThreadPoolExecutor` — ordering may be non-deterministic between runs

## 6. Event Ordering

### Potential Out-of-Order Processing
1. `handle_event()` returns an action string ("click:x,y")
2. Main loop processes the action and calls `_do_left_click(x, y)`
3. Same iteration, the game state checks `engine.state` — but this is `AttributeError` (CRITICAL BUG)

If the AttributeError were fixed, the ordering would be:
- Event → action → engine mutation → state check → draw
- This is correct single-threaded ordering with no race conditions

## 7. Win/Loss Condition Completeness

### Win Condition Gap (FIND-STATE-MEDIUM-m007a)
Current: `revealed_count == total_safe` → "won"
Required per GAME_DESIGN.md: ALL mines flagged AND all safes revealed AND no wrong flags

The current win condition ignores the image-reconstruction mechanic entirely. A player can "win" without placing a single flag.

### Loss Condition
Current: `board.reveal(x, y)` where cell is mine → "lost" + all mines revealed
This is correct per standard Minesweeper rules. No issues found.

### Chording Safety
`board.chord()` correctly checks `flagged_count == number` before revealing. Safe cells cannot be chord-revealed accidentally.
